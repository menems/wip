## 1. Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `PORT` | `8080` | TCP port the HTTP server listens on. |
| `OTEL_SERVICE_NAME` | `sass` | Service name embedded in every trace span's resource attributes. |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | _(unset)_ | OTLP HTTP collector URL (e.g. `http://jaeger:4318`). When unset, spans are printed to stdout in a human-readable format. |

### Notes

- `PORT` must be a valid port number. The server binds to `:<PORT>` (all interfaces).
- All standard `OTEL_*` SDK environment variables are honoured by the
  OpenTelemetry Go SDK (e.g. `OTEL_RESOURCE_ATTRIBUTES`, `OTEL_PROPAGATORS`).
  See the [OTel environment variable spec](https://opentelemetry.io/docs/specs/otel/configuration/sdk-environment-variables/)
  for the full list.

## 2. Metrics Catalogue

All metrics are exposed on `GET /metrics` in the Prometheus text format
(version 0.0.4). Each series carries `method` and `path` labels.

### `http_requests_total`

| Field | Value |
|-------|-------|
| Type | Counter |
| Unit | requests |
| Labels | `method`, `path` |
| Help | Total number of HTTP requests completed. |

Incremented once per request **after** the handler returns. Use this series
for request rate (QPS) and error ratio alerts.

```promql
# Request rate over 5 minutes
rate(http_requests_total[5m])
```

---

### `http_request_duration_seconds`

| Field | Value |
|-------|-------|
| Type | Histogram |
| Unit | seconds |
| Labels | `method`, `path` |
| Buckets | Prometheus default: 0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10 |
| Help | Duration of HTTP requests in seconds. |

Observed once per request after the handler returns. Use this for latency
percentile alerts (p50, p95, p99).

```promql
# 95th-percentile latency over 5 minutes
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
```

---

### `http_requests_in_flight`

| Field | Value |
|-------|-------|
| Type | Gauge |
| Unit | requests |
| Labels | `method`, `path` |
| Help | Number of HTTP requests currently being served. |

Incremented when a request enters the middleware and decremented when the
handler returns. Use this to detect connection spikes or goroutine leaks.

```promql
# Requests in flight right now
http_requests_in_flight
```

---

### Label Definitions

| Label | Source | Example values |
|-------|--------|----------------|
| `method` | `r.Method` | `GET`, `POST`, `DELETE` |
| `path` | `r.URL.Path` | `/health/live`, `/metrics`, `/` |

> **High-cardinality warning:** `path` is taken verbatim from the request URL.
> Avoid exposing the server to URLs with high-cardinality path segments
> (e.g. `/users/{id}`) without first adding a route-normalisation middleware.

## 3. Log Field Reference

Every handled HTTP request produces exactly one structured JSON log line emitted
to **stdout** at level `INFO`. The format is set by `log/slog` with
`slog.NewJSONHandler`.

### Log Line Shape

```json
{
  "time":       "2026-02-27T12:34:56.789Z",
  "level":      "INFO",
  "msg":        "request",
  "method":     "GET",
  "path":       "/health/live",
  "status":     200,
  "duration_ms": 3,
  "request_id": "abc-123-def-456"
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| `time` | RFC 3339 string | Timestamp when the log line was emitted (after the handler returns). |
| `level` | string | Always `INFO` for request logs. |
| `msg` | string | Always `"request"`. |
| `method` | string | HTTP request method (`GET`, `POST`, …). |
| `path` | string | Request URL path (no query string). |
| `status` | integer | HTTP response status code. Defaults to `200` if the handler never called `WriteHeader`. |
| `duration_ms` | integer | Wall-clock time in milliseconds from request start to handler completion. |
| `request_id` | string | Value of the `X-Request-Id` header — generated by the server if not supplied by the client. Matches the span's `request_id` attribute. |

### Correlating Logs and Traces

The `request_id` field is the join key between a log line and its distributed
trace span. Given a `request_id` from a log:

```bash
# Find the matching span in Jaeger
curl "http://localhost:16686/api/traces?service=sass&tags=%7B%22request_id%22%3A%22<REQUEST_ID>%22%7D"
```

## 4. Local Tracing Setup (Jaeger)

Use Jaeger all-in-one to visualise traces during local development without
a full observability stack.

### Step 1 — Start Jaeger

```bash
docker run --rm -d \
  --name jaeger \
  -p 16686:16686 \
  -p 4318:4318 \
  jaegertracing/all-in-one:latest
```

| Port | Protocol | Purpose |
|------|----------|---------|
| `16686` | HTTP | Jaeger UI |
| `4318` | HTTP | OTLP HTTP receiver (spans) |

### Step 2 — Start the Server

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318 \
OTEL_SERVICE_NAME=sass \
go run .
```

Or with a pre-built binary:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318 \
OTEL_SERVICE_NAME=sass \
./server
```

### Step 3 — Generate Some Traffic

```bash
curl http://localhost:8080/health/live
curl http://localhost:8080/health/ready
curl http://localhost:8080/metrics
```

### Step 4 — Inspect Traces

Open the Jaeger UI at **http://localhost:16686**, select the `sass` service,
and click **Find Traces**. Each HTTP request appears as a single root span
named `METHOD /path` with a `request_id` attribute.

### Teardown

```bash
docker stop jaeger
```

## 5. Health Probe Reference

### `GET /health/live` — Liveness

| | |
|-|-|
| **Purpose** | Kubernetes liveness probe — decides whether to restart the container. |
| **Behaviour** | Always returns `200 OK`. Never runs application checks. |
| **Response body** | `{"status":"ok"}` (`application/json`) |

Configure in a Pod spec:

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
```

---

### `GET /health/ready` — Readiness

| | |
|-|-|
| **Purpose** | Kubernetes readiness probe — decides whether to send traffic. |
| **Behaviour** | Runs all registered `Checker` implementations concurrently. |
| **Success** | All checkers pass → `200 OK`, body `{"status":"ok"}` (`application/json`) |
| **Failure** | Any checker fails → `503 Service Unavailable`, RFC 7807 body (`application/problem+json`) |

#### 503 Response Body

```json
{
  "type":       "about:blank",
  "title":      "Service Unavailable",
  "status":     503,
  "detail":     "db connection refused",
  "request_id": "abc-123"
}
```

Configure in a Pod spec:

```yaml
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 3
```
